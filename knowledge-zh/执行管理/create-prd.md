# 创建产品需求文档

## 目的

您是一位经验丰富的产品经理，负责为 $ARGUMENTS 创建全面的产品需求文档（PRD）。本文档将作为您的产品或功能的权威规范，使利益相关者保持一致并指导开发。

## 上下文

结构良好的 PRD 清晰地传达了产品计划的内容、原因和方式。此技能使用经过验证的 8 节模板，可有效地向工程师、设计师、领导和利益相关者传达产品愿景。

## 说明

1. **收集信息**：如果用户提供文件，请仔细阅读。如果他们提到研究、URL 或客户数据，请使用网络搜索收集额外的上下文和市场洞察。

2. **逐步思考**：在写作前，分析：
   - 我们要解决什么问题？
   - 我们为谁解决这个问题？
   - 我们如何衡量成功？
   - 我们的约束和假设是什么？

3. **应用 PRD 模板**：创建包含以下 8 个部分的文档：

   **1. 摘要**（2-3 句话）
   - 本文档是关于什么的？

   **2. 联系人**
   - 主要利益相关者的姓名、角色和评论

   **3. 背景**
   - 上下文：此计划是关于什么的？
   - 为什么是现在？有什么变化吗？
   - 这是最近才成为可能的事情吗？

   **4. 目标**
   - 目标是什么？为什么重要？
   - 它将如何使公司和客户受益？
   - 它如何与愿景和战略保持一致？
   - 关键结果：你将如何衡量成功？（使用 SMART OKR 格式）

   **5. 市场细分**
   - 我们为谁构建这个？
   - 存在什么约束？
   - 注意：市场是由人们的问题/工作定义的，而不是人口统计数据

   **6. 价值主张**
   - 我们要解决哪些客户工作/需求？
   - 客户将获得什么？
   - 他们将避免哪些痛苦？
   - 哪些问题我们解决得比竞争对手更好？
   - 考虑价值曲线框架

   **7. 解决方案**
   - 7.1 用户体验/原型（线框图、用户流程）
   - 7.2 关键功能（详细功能描述）
   - 7.3 技术（可选，仅在相关时）
   - 7.4 假设（我们相信但尚未证明的内容）

   **8. 发布**
   - 可能需要多长时间？
   - 第一版与未来版本包含什么？
   - 避免确切日期；使用相对时间框架

4. **使用易懂的语言**：为小学毕业生写作。避免行话。使用清晰、简短的句子。

5. **结构化输出**：将 PRD 呈现为格式良好的 Markdown 文档，带有清晰的标题和部分。

6. **保存输出**：如果 PRD 内容丰富（通常会如此），将其保存为 Markdown 文档，格式为：`PRD-[产品名称].md`

## 注意事项

- 尽可能具体并以数据为驱动
- 将每个部分与整体战略联系起来
- 清楚地标记假设，以便团队可以验证它们
- 保持文档简洁但完整

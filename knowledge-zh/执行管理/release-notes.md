## 发布说明生成器

将技术工单、PRD 或内部变更日志转换为 polished、面向用户的发布说明。

### 上下文

您正在为 **$ARGUMENTS** 编写发布说明。

如果用户提供文件（JIRA 导出、Linear 工单、PRD、Git 日志或内部变更日志），请先阅读它们。如果他们提到产品 URL，请使用网络搜索了解产品和受众。

### 说明

1. **收集原始材料**：阅读所有提供的工单、变更日志或描述。提取：
   - 发生了什么变化（功能、改进或修复）
   - 影响谁（哪个用户细分）
   - 为什么重要（用户 benefit）

2. **分类变更**：
   - **新功能**：全新的功能
   - **改进**：对现有功能的增强
   - **Bug 修复**：已解决的问题
   - **重大变更**：任何需要用户行动的内容（迁移、API 变更）
   - **弃用**：即将停止的功能

3. **编写每个条目**，遵循以下原则：
   - 以用户 benefit 开头，而非技术变更
   - 使用通俗易懂的语言 — 避免行话、内部代号或工单编号
   - 每个条目保持 1-3 句话
   - 如果用户提供，包括视觉效果或截图

   **示例转换**：
   - 技术："为仪表板 API 端点实现 Redis 缓存层"
   - 面向用户："仪表板现在加载速度快达 3 倍，因此您花在等待上的时间更少，分析的时间更多。"

   - 技术："修复并发结账流程中的竞争条件"
   - 面向用户："修复了在高流量期间某些订单可能失败的问题。"

4. **结构化发布说明**：

   ```
   # [产品名称] — [版本 / 日期]

   ## 新功能
   - **[功能名称]**：[1-2 句描述它的作用和为什么重要]

   ## 改进
   - **[领域]**：[什么变好了以及它如何帮助]

   ## Bug 修复
   - 修复了 [用户术语中的问题描述]

   ## 重大变更（如有）
   - **需要操作**：[用户需要做什么]
   ```

5. **调整语气**以匹配产品的声音 — B2B 专业，消费者友好，API 以开发者为中心。

保存为 Markdown 文档。如果用户需要 HTML 或其他格式，请相应转换。
